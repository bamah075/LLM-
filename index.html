CRITICAL: Most local LLM deployments are insecure by default. Exposed APIs, unverified model weights, and missing prompt injection defenses create serious attack surfaces.

Model Integrity & Verification

Threat: GGUF files can contain malicious chat templates that hijack model behavior. Model weights can be tampered with post-download.

Verification Checklist

# Verify GGUF checksum sha256sum llama-3.3-70b.Q4_K_M.gguf # Compare against published hash on Hugging Face model card # Inspect chat template inside GGUF python3 -c "from gguf import GGUFReader; r=GGUFReader('model.gguf'); print(r.metadata['tokenizer.chat_template'])"

API Authentication & Access Control

Threat: Unauthenticated inference endpoints allow anyone with network access to exhaust GPU resources or extract model behavior.

Required Controls

# nginx.conf example for API gateway location /v1/chat/completions { auth_request /auth; limit_req_zone $binary_remote_addr zone=llm:10m rate=10r/m; proxy_pass http://llm-backend:8000; }

Container Hardening

Default Docker settings leave significant attack surface. Harden containers before production.
services: llm-inference: image: vllm/vllm-openai:v0.4.2 # PINNED user: "1000:1000" read_only: true cap_drop: [ALL] cap_add: [SYS_NICE] security_opt: - no-new-privileges:true - seccomp:./seccomp-profile.json networks: - inference_internal # internal: true

PRODUCTION READINESS: A demo is not production. You need CI/CD, health checks, rollback plans, and runbooks before serving real users.

CI/CD Pipeline

# .github/workflows/deploy.yml name: Deploy LLM Inference on: push: branches: [main] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Model Integrity Check run: | sha256sum -c models/SHA256SUMS python scripts/verify_chat_template.py - name: Load Test run: locust -f tests/load_test.py --host http://staging - name: Safety Test run: pytest tests/test_prompt_injection.py deploy-staging: needs: test steps: - name: Blue-Green Deploy run: | docker-compose -f docker-compose.staging.yml up -d --no-deps llm-green curl -f http://staging/health || exit 1 ./scripts/switch-traffic.sh green deploy-production: needs: deploy-staging environment: production steps: - name: Canary Deploy (10%) run: ./scripts/canary-deploy.sh 10 - name: Monitor for 30min run: ./scripts/wait-for-metrics.sh --p95-latency < 5s --error-rate < 0.1% - name: Full Rollout run: ./scripts/canary-deploy.sh 100

Health Checks & Graceful Shutdown

# Health check endpoint @app.get("/health") async def health(): return { "status": "healthy", "model_loaded": model.is_loaded(), "gpu_available": torch.cuda.is_available(), "vram_free_gb": get_free_vram(), "queue_depth": request_queue.size(), "last_inference": time_since_last_request() } # Graceful shutdown handler @app.on_event("shutdown") async def shutdown(): # Stop accepting new requests request_queue.close() # Wait for in-flight requests await request_queue.drain(timeout=30) # Save KV cache if needed model.save_kv_cache() # Release GPU memory torch.cuda.empty_cache()

Runbook: Common Failure Modes

GPU OOM (Out of Memory)

# Symptoms: Requests fail with CUDA OOM # Diagnosis: nvidia-smi # Check VRAM usage dmesg | grep -i oom # Kernel OOM logs # Remediation: 1. Reduce MAX_MODEL_LEN in config 2. Enable KV cache quantization 3. Reduce batch size 4. Offload more layers to CPU RAM 5. Restart inference container

Model Corruption / Load Failure

# Symptoms: Model fails to load, checksum mismatch # Diagnosis: sha256sum model.gguf # Verify integrity ls -la /models/ # Check file sizes # Remediation: 1. Re-download from verified source 2. Check disk health (smartctl) 3. Verify chat template matches official 4. Load previous known-good version 5. Alert on-call for manual verification

Queue Saturation

# Symptoms: Latency spikes, timeout errors # Diagnosis: curl http://localhost:8000/metrics | grep queue_depth # Remediation: 1. Enable auto-scaling (add GPU nodes) 2. Route excess to cloud fallback 3. Implement request prioritization 4. Reduce context length temporarily 5. Scale down non-critical workloads

Driver / CUDA Crash

# Symptoms: GPU disappears, NVML errors # Diagnosis: nvidia-smi # Check driver status journalctl -u nvidia-persistenced # Driver logs # Remediation: 1. Restart nvidia-persistenced service 2. Reload NVIDIA kernel module 3. If persistent: downgrade driver version 4. Failover to standby GPU/node 5. Schedule maintenance window

Priority Checklist

Maximize Model Size

HIGHEST IMPACT

A 70B Q4 model beats a 7B Q8 model. Prioritize parameter count over quantization precision. Use layer offloading to RAM if VRAM is insufficient.

Implement RAG

VERY HIGH IMPACT

Inject relevant context from your documents. Use BGE-M3 for embeddings, ChromaDB for storage, and always add a reranker for top-5 chunk selection.

Domain Fine-Tuning

HIGH IMPACT

Use QLoRA (16-24GB VRAM) with 1,000+ domain examples. A fine-tuned 7B often beats a base 70B on narrow tasks.

Security Hardening

CRITICAL FOR PRODUCTION

Verify model checksums, implement JWT auth, harden containers, add prompt injection defenses. Without this, you are one breach away from disaster.

Monitoring & Alerting

CRITICAL FOR PRODUCTION

Track TTFT, TPS, queue depth, VRAM usage. Alert on OOM, overheating, error spikes. You cannot optimize what you cannot measure.

Inference Optimization

MEDIUM IMPACT

Enable FlashAttention-3, use speculative decoding with a 1B draft model, and quantize KV cache. These give 2-3x speedup with zero quality loss.

Use MoE Models

MEDIUM IMPACT

Mixtral 8x7B and Qwen 3 Coder use sparse activation. Load only active experts in VRAM for 70B+ quality at ~13B compute cost.

Pro Tip Use llama.cpp with --tensor-split and --override-tensor to apply different quants per layer type. This preserves 90%+ of FP16 quality at 60% of the size.

Fine-Tuning Methods Comparison

QLoRA

16-24GB VRAM | Best for most users

4-bit base model + small adapter matrices (LoRA). Train only 1-2% of parameters. Merge adapters after training. Use Unsloth for 2x speed.

Recommended

LoRA

24-48GB VRAM | Higher quality

8-bit base model + adapters. Better precision than QLoRA. Good for larger datasets (>10K examples) where QLoRA may underfit.

Full Fine-Tune

80GB+ VRAM | Maximum quality

Update all model parameters. Requires multi-GPU or A100/H100. Only worth it for enterprise with massive proprietary datasets.

Hybrid Local + Cloud Architecture

When to Use Hybrid

  • Sensitive data -> Local model
  • Complex reasoning -> Cloud fallback
  • Cost optimization -> Local for 90% of queries
  • Redundancy -> Local backup when cloud is down

Routing Logic

# Simple quality gate router if contains_pii(query) or sensitivity == "high": route_to_local() elif task_complexity_score(query) > 0.8: route_to_cloud() else: route_to_local() # default

5-Minute Quick Start

Step 1: Install Ollama

# macOS / Linux curl -fsSL https://ollama.com/install.sh | sh # Windows: Download from ollama.com

Step 2: Run Your First Model

# General purpose (recommended) ollama run qwen3:8b # Low resource hardware ollama run phi4-mini # Coding focused ollama run qwen3:7b

Step 3: Test the API

curl http://localhost:11434/api/generate -d '{ "model": "qwen3:8b", "prompt": "Explain quantum computing" }'

Step 4: Set Up RAG (Python)

pip install chromadb sentence-transformers # embed_documents.py from chromadb import Client from sentence_transformers import SentenceTransformer client = Client() collection = client.create_collection("docs") model = SentenceTransformer("BAAI/bge-m3") docs = ["Your document text here..."] embeddings = model.encode(docs) collection.add(embeddings=embeddings, documents=docs, ids=["doc1"])

Step 5: Query with RAG

query = "Your question?" q_embed = model.encode([query]) results = collection.query(query_embeddings=q_embed, n_results=3) # Inject results into prompt
